# GitHub Desktop Private/Public 관리 체크리스트


# 🔹 1. 저장소 연결 관리
## Desktop에서 Repository 이름 확인

작업 중인 저장소가 정확히 Private/Public인지 매번 확인

Remote URL 확인

## Repository → Repository Settings → Remote

### → URL이 올바른지, 실수로 다른 저장소에 연결 안 됐는지 체크

## 폴더와 저장소 일대일 매칭
F:\momo\momo1 → Public
F:\momo\momo2 → Private
## → 절대로 섞어 쓰지 않기



# 🔹 2. 커밋 & 푸시 주의

커밋 전 항상 Changes 확인
민감 정보 포함 여부
테스트용 파일이 Public에 들어가는지 확인
커밋 메시지 스타일
Public: 깔끔, 배포용, 민감 정보 없음
Private: 자유, 실험용
푸시 전에 Repository 확인
Desktop 상단 Repository 표시
실수로 다른 repo에 Push 방지



# 🔹 3. 브랜치 관리

Private: 자유롭게 feature 브랜치, 임시 브랜치 생성/삭제
Public: main/master 중심 + 배포용 브랜치만
브랜치 머지 시, Private → Public 이동은 필요한 파일만 복사 후 커밋



# 🔹 4. 파일 관리 & .gitignore

Public에는 절대로 개인 정보, 로그, 환경 설정 파일 포함 금지
Private는 자유롭게 관리
동일한 프로젝트라도 Private/Public 별 .gitignore 관리



# 🔹 5. 깃 충돌(Conflict) 주의

같은 파일을 Private에서 수정 → Public에서 수정 → Push 시 충돌 발생 가능
Desktop에서 충돌 발생하면 항상 Changes 확인 후 Resolve
충돌 해결 과정에서 실수로 민감 정보 Public에 포함되지 않게 주의



# 🔹 6. Backup & 실수 대비

중요한 Private repo는 정기 백업 (로컬 복사 or 다른 Remote)
Public repo는 필요 시 Release Tag 활용
“혹시나” 실수로 Push한 경우
GitHub → Settings → Repository → Change visibility 또는 force push 조심



# 🔹 7. 습관화 포인트 (핵심)

Repository 이름/Private-Public 표시 항상 확인

커밋 전 Changes 확인
브랜치 전략에 따라 Push
파일 유형/민감 정보 체크
필요 시 로컬/Remote Backup
